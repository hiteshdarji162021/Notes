## Why learn?

## 1Ô∏è‚É£ `getByText()`

### Code
```ts
await page.getByText('Register Account', { exact: true }).highlight();
```

### Output
- The **"Register Account"** header is highlighted

### Use case
- Validate **visible static text**
- Headers, buttons, labels

### Playwright note
- Matches **only visible text**
- `exact: true` avoids partial match

---

## 2Ô∏è‚É£ `locator()` with `hasText`

### Code
```ts
page.locator('h1', { hasText: 'Register Account' }).highlight();
```

### Output
- `<h1>` containing text is highlighted

### Use case
- When multiple tags exist but text is unique
- Useful in **tables, lists, repeated tags**

---

## 3Ô∏è‚É£ `tag:text()` selector

### Code
```ts
page.locator('h1:text("Register Account")').highlight();
```

### Output
- Target `<h1>` is highlighted

### Use case
- Compact selector
- Quick debugging

‚ö†Ô∏è Less readable, avoid for large suites

---

## üîü CSS Selectors (Quick Reference)

### 1Ô∏è‚É£ ID Selector

### Selector
```css
#input-email
input#input-email
```

### Description
- Targets an element using its **unique `id` attribute**.
- Fastest and most reliable CSS selector.

### When to use
- When `id` is **unique and stable**
- Login forms, critical inputs, buttons

### Playwright example
```ts
await page.locator('#input-email').fill('user@gmail.com');
```

---

### 2Ô∏è‚É£ Class Selector

### Selector
```css
.form-control
input.form-control
```

### Description
- Targets elements using **CSS class name**.
- Class names **can be reused** across elements.

### When to use
- When no stable id exists
- Styling-based elements

‚ö†Ô∏è Risky if class is reused widely

### Playwright example
```ts
await page.locator('input.form-control').fill('test');
```

---

### 3Ô∏è‚É£ Multiple Class Selector

### Selector
```css
input.form-control.input-lg
```

### Description
- Targets elements having **multiple specific classes**.
- More precise than single class selector.

### When to use
- When single class matches many elements
- UI frameworks like Bootstrap

### Playwright example
```ts
await page.locator('input.form-control.input-lg').fill('test');
```

---

### 4Ô∏è‚É£ Attribute Selector

### Selector
```css
input[name='email']
input[name='email'][type='email']
```

### Description
- Targets elements using **HTML attributes**.
- Very readable and stable when attributes are meaningful.

### When to use
- Form fields
- Inputs without ids

### Playwright example
```ts
await page.locator("input[name='email']").fill('user@gmail.com');
```

---

### 5Ô∏è‚É£ Contains (`*=`) Selector

### Selector
```css
input[placeholder*='Address']
img[src*='logo']
```

### Description
- Matches elements where attribute **contains a substring**.

### When to use
- Dynamic attributes
- Partial matching required

‚ö†Ô∏è Use carefully ‚Äî can become flaky

### Playwright example
```ts
await page.locator("img[src*='logo']").click();
```

---

### 6Ô∏è‚É£ Starts With (`^=`) and Ends With (`$=`)

### Selector
```css
img[alt^='naveen']
img[alt$='opencart']
```

### Description
- `^=` matches attribute **starting with** value
- `$=` matches attribute **ending with** value

### When to use
- Predictable attribute patterns
- Dynamic ids or image names

### Playwright example
```ts
await page.locator("img[alt$='opencart']").click();
```

---

### 7Ô∏è‚É£ Parent ‚Üí Child Selector

### Selector
```css
div#content input#input-email
div.form-group > input#input-email
```

### Description
- Space (` `) ‚Üí any descendant
- `>` ‚Üí direct child only

### When to use
- Narrow down elements inside specific sections
- Avoid collisions in large DOMs

### Playwright example
```ts
await page.locator('div.form-group > input#input-email').fill('test');
```

---

## ‚úÖ Best Practices (IMPORTANT)

- Prefer **ID and attribute selectors** first
- Avoid over-long CSS chains
- Avoid dynamic classes
- Prefer `getByTestId()` over complex CSS

---

## Architect Golden Rule üèÜ

> **CSS selectors are powerful, but should be your fallback‚Äînot your first choice.**

---


## ‚úÖ Best Practices (MANDATORY)

- Prefer `getByRole()` / `getByText()` for user-visible actions
- Use `hasText` for filtering repeated elements
- Use `inputValue()` for form value validation
- Never click disabled elements
- Prefer semantic locators over complex CSS

---

## Architect Golden Rule üèÜ

> **Validate what the user sees and can interact with, not just what exists in the DOM.**
