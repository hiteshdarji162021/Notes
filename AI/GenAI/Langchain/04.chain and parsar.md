## 2. Manual Chaining (Sequential Prompts)

### 🧑‍🏫 Teacher Explanation

Imagine you're writing a research paper. First you write a **detailed report**, then someone else **summarizes it**. That's sequential chaining — the output of Step 1 becomes the input of Step 2.

In the manual approach, you do this yourself: call the model, grab the result, pass it to the next prompt by hand.

### 🧠 Mental Model

```
topic ("black hole")
       │
       ▼
  template1 ──► "Write a detailed report on black hole"
       │
       ▼
    model ──► [Long detailed text about black holes]
       │
       ▼
  template2 ──► "Write a 5 line summary on the following text. \n [long text]"
       │
       ▼
    model ──► [5-line summary]
       │
       ▼
   print output
```

### 🔧 Tech Example (Step-by-Step)

```python
# Step 1: Build prompt 1
prompt1 = template1.invoke({'topic': 'black hole'})
# Result: StringPromptValue with "Write a detailed report on black hole"

# Step 2: Send to model → get detailed text
result = model.invoke(prompt1)
# result is an AIMessage; access text via result.content

# Step 3: Build prompt 2 using Step 2's output
prompt2 = template2.invoke({'text': result.content})

# Step 4: Get the summary
result1 = model.invoke(prompt2)
print(result1.content)
```

### ⚠️ v0.3 → v1.0+ Changes

| What | v0.3 | v1.0+ |
|------|------|--------|
| `PromptTemplate` import | `from langchain.prompts` | `from langchain_core.prompts` |
| `template.format()` | Common in v0.3 | Use `.invoke({})` — returns `PromptValue` |
| Output access | `result['text']` or `result` (string) | `result.content` — always `AIMessage` |
| `input_variables` | Required | Still required (same) |

### 📋 Bug Alert in Original Code

```python
# ❌ Bug in original code — uses /n (forward slash n), not a real newline
template='Write a 5 line summary on the following text. /n {text}'

# ✅ Correct
template='Write a 5 line summary on the following text.\n{text}'
```

### 💻 Full Code

```python
from langchain_huggingface import ChatHuggingFace, HuggingFaceEndpoint
from dotenv import load_dotenv
from langchain_core.prompts import PromptTemplate

load_dotenv()

# Model setup
llm = HuggingFaceEndpoint(
    repo_id="meta-llama/Llama-3.1-8B-Instruct",
    task="text-generation",
    provider="auto",   # HuggingFace auto-picks a working provider
    max_new_tokens=512
)
model = ChatHuggingFace(llm=llm)

# Prompt 1: Detailed report
template1 = PromptTemplate(
    template='Write a detailed report on {topic}',
    input_variables=['topic']
)

# Prompt 2: Summary
template2 = PromptTemplate(
    template='Write a 5 line summary on the following text.\n{text}',
    input_variables=['text']
)

# Manual chain execution
prompt1 = template1.invoke({'topic': 'black hole'})
result = model.invoke(prompt1)

prompt2 = template2.invoke({'text': result.content})
result1 = model.invoke(prompt2)

print(result1.content)
```

## 3. LCEL — The Pipe `|` Chain

### 🧑‍🏫 Teacher Explanation

LangChain Expression Language (LCEL) is LangChain's superpower. Instead of manually calling each step, you **connect them with pipes `|`** — just like Linux command-line pipes.

Think of an assembly line:
```
Raw material | Machine 1 | Machine 2 | Machine 3 | Final Product
```
Each machine takes what the previous one produced and transforms it further. You don't manage the flow — the line does it automatically.

### 🧠 Mental Model

```
 Input: {'topic': 'black hole'}
        │
        ▼
   template1         → formats the prompt
        │
        ▼
      model          → generates detailed report (AIMessage)
        │
        ▼
     parser          → extracts .content → plain string
        │
        ▼
   template2         → wraps string into summary prompt
        │
        ▼
      model          → generates 5-line summary
        │
        ▼
     parser          → extracts .content → plain string
        │
        ▼
   Final Output: "Black holes are regions of spacetime..."
```

### 🔧 Tech Example

```python
from langchain_core.output_parsers import StrOutputParser

parser = StrOutputParser()

# The entire pipeline in one line
chain = template1 | model | parser | template2 | model | parser

# One call to rule them all
result = chain.invoke({'topic': 'black hole'})
print(result)
```

The `StrOutputParser` converts `AIMessage` → `str`, which is needed between the two templates so `template2` receives a plain string for its `{text}` variable.

### ⚠️ v0.3 → v1.0+ Changes

| What | v0.3 | v1.0+ |
|------|------|--------|
| LCEL pipe `|` syntax | ❌ Not available | ✅ Core feature — use this always |
| `LLMChain` | ✅ Main way to chain | ❌ Deprecated — use LCEL `|` instead |
| `SequentialChain` | ✅ Used for multi-step | ❌ Deprecated — use LCEL `|` instead |
| `StrOutputParser` | `from langchain.schema` | `from langchain_core.output_parsers` |
| Streaming | Complex | ✅ Built-in with `.stream()` |
| Async | Complex | ✅ Built-in with `.ainvoke()` |

### 💡 LCEL Bonus Powers

```python
# Async (non-blocking)
result = await chain.ainvoke({'topic': 'black hole'})

# Streaming (word by word output)
for chunk in chain.stream({'topic': 'black hole'}):
    print(chunk, end='', flush=True)

# Batch (multiple inputs at once)
results = chain.batch([
    {'topic': 'black hole'},
    {'topic': 'neutron star'},
])
```

### 💻 Full Code

```python
from langchain_huggingface import ChatHuggingFace, HuggingFaceEndpoint
from dotenv import load_dotenv
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser

load_dotenv()

llm = HuggingFaceEndpoint(
    repo_id="meta-llama/Llama-3.1-8B-Instruct",
    task="text-generation",
    provider="auto",
    max_new_tokens=512
)
model = ChatHuggingFace(llm=llm)

template1 = PromptTemplate(
    template='Write a detailed report on {topic}',
    input_variables=['topic']
)

template2 = PromptTemplate(
    template='Write a 5 line summary on the following text.\n{text}',
    input_variables=['text']
)

parser = StrOutputParser()

# LCEL chain: connect everything with pipes
chain = template1 | model | parser | template2 | model | parser

result = chain.invoke({'topic': 'black hole'})
print(result)
```

## 4. JSON Output Parser

### 🧑‍🏫 Teacher Explanation

By default, the model returns free-form text. But what if you want **structured data** — like a Python dictionary or a list — that your application can process programmatically?

`JsonOutputParser` tells the model: *"Hey, reply ONLY in JSON format."* It injects formatting instructions into your prompt automatically, then parses the model's response back into a Python dict.

Think of it as ordering from a restaurant with a specific order form instead of just saying "give me something good." You get back exactly what you asked for, in the format you need.

### 🧠 Mental Model

```
parser.get_format_instructions()
       │  "Return a valid JSON object..."
       ▼
  PromptTemplate (with {format_instruction})
       │  "Give me 5 facts about black holes.
       │   Return a valid JSON object..."
       ▼
     model
       │  '{"fact1": "Black holes...", "fact2": "..."}'
       ▼
  JsonOutputParser
       │  Parses JSON string → Python dict
       ▼
  {'fact1': 'Black holes...', 'fact2': '...'}
```

### 🔧 Tech Example

```python
from langchain_core.output_parsers import JsonOutputParser

parser = JsonOutputParser()

# See what instructions the parser injects
print(parser.get_format_instructions())
# Output: "Return a markdown code snippet with a JSON object formatted
#          to look like: {"key": value}"

template = PromptTemplate(
    template='Give me 5 facts about {topic}\n{format_instruction}',
    input_variables=['topic'],
    partial_variables={'format_instruction': parser.get_format_instructions()}
)

chain = template | model | parser
result = chain.invoke({'topic': 'black hole'})

# result is now a Python dict, not a string!
print(type(result))   # <class 'dict'>
print(result['fact1'])
```

### ⚠️ v0.3 → v1.0+ Changes

| What | v0.3 | v1.0+ |
|------|------|--------|
| `JsonOutputParser` import | `from langchain.output_parsers` | `from langchain_core.output_parsers` |
| `partial_variables` in `PromptTemplate` | ✅ Available | ✅ Same — no change |
| Output type | Similar | Returns `dict` or `list` — true Python types |
| Error handling | Manual | `.with_retry()` and `.with_fallbacks()` available |

### 💡 When to Use

- You need a **list** of items to loop over
- You need **key-value** data to pass to another function
- You're building an API that returns structured JSON
- You want **predictable, parseable** output

### 💻 Full Code

```python
from langchain_huggingface import ChatHuggingFace, HuggingFaceEndpoint
from dotenv import load_dotenv
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import JsonOutputParser

load_dotenv()

llm = HuggingFaceEndpoint(
    repo_id="meta-llama/Llama-3.1-8B-Instruct",
    task="text-generation"
)
model = ChatHuggingFace(llm=llm)

parser = JsonOutputParser()

template = PromptTemplate(
    template='Give me 5 facts about {topic}\n{format_instruction}',
    input_variables=['topic'],
    partial_variables={'format_instruction': parser.get_format_instructions()}
)

chain = template | model | parser

result = chain.invoke({'topic': 'black hole'})
print(result)
# Example output: {'fact1': 'Black holes...', 'fact2': '...', ...}
```

## 5. Structured Output Parser

### 🧑‍🏫 Teacher Explanation

`JsonOutputParser` is flexible but loose — the model decides the JSON shape. `StructuredOutputParser` lets you **define the exact schema** upfront using `ResponseSchema` objects.

Think of it like designing a form with labeled fields. You say: *"I want exactly these 3 fields: fact_1, fact_2, fact_3."* The model fills in the form; the parser returns exactly those fields.

### 🧠 Mental Model

```
You define the schema (the form):
  ┌───────────────────────────────────────┐
  │ fact_1: "Fact 1 about the topic"      │
  │ fact_2: "Fact 2 about the topic"      │
  │ fact_3: "Fact 3 about the topic"      │
  └───────────────────────────────────────┘
           │
           ▼
  Parser generates instructions for the model
           │
           ▼
       Model fills in the form
           │
           ▼
  Parser returns: {'fact_1': '...', 'fact_2': '...', 'fact_3': '...'}
```

### 🔧 Tech Example

```python
from langchain_core.output_parsers import StructuredOutputParser, ResponseSchema

schema = [
    ResponseSchema(name='fact_1', description='Fact 1 about the topic'),
    ResponseSchema(name='fact_2', description='Fact 2 about the topic'),
    ResponseSchema(name='fact_3', description='Fact 3 about the topic'),
]

parser = StructuredOutputParser.from_response_schemas(schema)

# See what instructions get injected
print(parser.get_format_instructions())
```

### ⚠️ v0.3 → v1.0+ Changes — IMPORTANT BREAKING CHANGE

```python
# ❌ WRONG — as in original code (langchain_classic doesn't exist)
from langchain_classic.output_parsers import StructuredOutputParser, ResponseSchema

# ✅ CORRECT for v1.0+
from langchain_core.output_parsers import StructuredOutputParser, ResponseSchema

# If using older community package:
# from langchain.output_parsers import StructuredOutputParser, ResponseSchema
```

| What | v0.3 | v1.0+ |
|------|------|--------|
| Import path | `from langchain.output_parsers` | `from langchain_core.output_parsers` |
| `langchain_classic` | ❌ Does not exist | ❌ Does not exist — this was a bug in original code |
| `ResponseSchema` | Same usage | Same usage |
| Schema validation | No type enforcement | Use Pydantic parser for strict types |

### 💡 StructuredOutputParser vs JsonOutputParser

| Feature | `JsonOutputParser` | `StructuredOutputParser` |
|---------|-------------------|--------------------------|
| Schema defined upfront | ❌ | ✅ |
| Field descriptions | ❌ | ✅ (helps model understand) |
| Output type | `dict` / `list` | `dict` with exact keys |
| Best for | Flexible JSON | Known fixed schema |

### 💻 Full Code

```python
from langchain_huggingface import ChatHuggingFace, HuggingFaceEndpoint
from dotenv import load_dotenv
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StructuredOutputParser, ResponseSchema

load_dotenv()

llm = HuggingFaceEndpoint(
    repo_id="meta-llama/Llama-3.1-8B-Instruct",
    task="text-generation"
)
model = ChatHuggingFace(llm=llm)

# Define the exact schema you want
schema = [
    ResponseSchema(name='fact_1', description='Fact 1 about the topic'),
    ResponseSchema(name='fact_2', description='Fact 2 about the topic'),
    ResponseSchema(name='fact_3', description='Fact 3 about the topic'),
]

parser = StructuredOutputParser.from_response_schemas(schema)

template = PromptTemplate(
    template='Give 3 facts about {topic}\n{format_instruction}',
    input_variables=['topic'],
    partial_variables={'format_instruction': parser.get_format_instructions()}
)

chain = template | model | parser

result = chain.invoke({'topic': 'black hole'})
print(result)
# Output: {'fact_1': '...', 'fact_2': '...', 'fact_3': '...'}
```

## 6. Pydantic Output Parser

### 🧑‍🏫 Teacher Explanation

`PydanticOutputParser` is the most powerful parser. Instead of a loose dict, you get back a **fully validated Python object** with types, constraints, and auto-completion in your IDE.

It uses **Pydantic** — Python's data validation library. You define a class like `Person` with fields and rules (e.g., `age > 18`), and LangChain + Pydantic will:
1. Tell the model exactly what JSON to return
2. Parse the JSON
3. Validate all fields and types
4. Return a proper `Person` object — not a raw dict

Think of it as not just getting a form back, but getting a fully checked, validated, typed Python object ready to use in your code.

### 🧠 Mental Model

```
Pydantic Model (your blueprint):
  class Person:
    name: str
    age: int  (must be > 18)
    city: str
         │
         ▼ parser auto-generates instructions
  "Return JSON with keys: name (string), age (int > 18), city (string)"
         │
         ▼ model responds
  '{"name": "Kumara", "age": 28, "city": "Colombo"}'
         │
         ▼ Pydantic validates & constructs
  Person(name='Kumara', age=28, city='Colombo')
         │
         ▼ you can access
  result.name  →  'Kumara'
  result.age   →  28
```

### 🔧 Tech Example

```python
from pydantic import BaseModel, Field
from langchain_core.output_parsers import PydanticOutputParser

class Person(BaseModel):
    name: str = Field(description='Name of the person')
    age: int = Field(gt=18, description='Age of the person')  # gt=18 means must be > 18
    city: str = Field(description='Name of the city the person belongs to')

parser = PydanticOutputParser(pydantic_object=Person)

# Access result as a proper typed object
result = chain.invoke({'place': 'sri lankan'})

print(result.name)    # 'Kumara'
print(result.age)     # 28
print(result.city)    # 'Colombo'
print(type(result))   # <class '__main__.Person'>
```

### ⚠️ v0.3 → v1.0+ Changes

| What | v0.3 | v1.0+ |
|------|------|--------|
| `PydanticOutputParser` import | `from langchain.output_parsers` | `from langchain_core.output_parsers` |
| Pydantic version | v1 (`validator`, `root_validator`) | ✅ Pydantic v2 (`model_validator`, `field_validator`) |
| `Field` constraints | `Field(gt=18)` | Same, but Pydantic v2 is stricter |
| Type coercion | Lenient | Strict by default in Pydantic v2 |

### Pydantic v1 vs v2 Field Validators

```python
# Pydantic v1 (used in older LangChain)
from pydantic import validator

class Person(BaseModel):
    age: int
    
    @validator('age')
    def age_must_be_adult(cls, v):
        if v <= 18:
            raise ValueError('Age must be > 18')
        return v

# Pydantic v2 (current — LangChain 1.0+)
from pydantic import field_validator

class Person(BaseModel):
    age: int
    
    @field_validator('age')
    @classmethod
    def age_must_be_adult(cls, v):
        if v <= 18:
            raise ValueError('Age must be > 18')
        return v
```

### 💡 Which Parser Should I Use?

```
Need output for further processing?
├── Free-form text?          → StrOutputParser
├── Any JSON shape?          → JsonOutputParser
├── Specific known fields?   → StructuredOutputParser
└── Typed, validated object? → PydanticOutputParser  ← Most powerful
```

### 💻 Full Code

```python
from langchain_huggingface import ChatHuggingFace, HuggingFaceEndpoint
from dotenv import load_dotenv
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import PydanticOutputParser
from pydantic import BaseModel, Field

load_dotenv()

llm = HuggingFaceEndpoint(
    repo_id="meta-llama/Llama-3.1-8B-Instruct",
    task="text-generation"
)
model = ChatHuggingFace(llm=llm)

# Define the Pydantic schema
class Person(BaseModel):
    name: str = Field(description='Name of the person')
    age: int = Field(gt=18, description='Age of the person')
    city: str = Field(description='Name of the city the person belongs to')

parser = PydanticOutputParser(pydantic_object=Person)

template = PromptTemplate(
    template='Generate the name, age and city of a fictional {place} person\n{format_instruction}',
    input_variables=['place'],
    partial_variables={'format_instruction': parser.get_format_instructions()}
)

chain = template | model | parser

result = chain.invoke({'place': 'sri lankan'})
print(result)
# Output: name='Kumara Perera' age=28 city='Colombo'
print(result.name)   # 'Kumara Perera'
print(result.age)    # 28
```

## 🏁 Key Takeaways

1. **LCEL `|` pipes replace all chain classes** — `LLMChain`, `SequentialChain` are gone
2. **All core imports moved to `langchain_core`** — stable, lightweight, no extras needed
3. **`langchain_classic` does not exist** — this was a bug in the original code
4. **Use `provider="auto"`** with `HuggingFaceEndpoint` to avoid provider errors
5. **`StrOutputParser` is the glue** between prompts in a multi-step pipeline
6. **Pydantic v2** is the default — update validators from `@validator` to `@field_validator`
7. **`result.content`** — always access model output this way (it returns `AIMessage`)
