## The Runnable Family — Big Picture

Before diving into code, understand the five primitives and what each one does:

```
┌─────────────────────────────────────────────────────────────────┐
│               LANGCHAIN RUNNABLE FAMILY                         │
├──────────────────────┬──────────────────────────────────────────┤
│  RunnableSequence    │  Run steps ONE AFTER ANOTHER             │
│                      │  Output of step N → input of step N+1    │
├──────────────────────┼──────────────────────────────────────────┤
│  RunnableParallel    │  Run multiple chains SIMULTANEOUSLY       │
│                      │  Same input → all branches → dict output  │
├──────────────────────┼──────────────────────────────────────────┤
│  RunnablePassthrough │  LET INPUT PASS UNCHANGED                 │
│                      │  Like a transparent pipe — data flows as-is│
├──────────────────────┼──────────────────────────────────────────┤
│  RunnableLambda      │  WRAP ANY PYTHON FUNCTION as a runnable   │
│                      │  Turn def word_count(text) into a step    │
├──────────────────────┼──────────────────────────────────────────┤
│  RunnableBranch      │  IF / ELIF / ELSE ROUTER                  │
│                      │  Pick one chain based on a condition      │
└──────────────────────┴──────────────────────────────────────────┘
```

**Mental model — think of a smart factory conveyor belt:**

```
Raw Material (your input dict)
        │
        ▼
  ┌─────────────────────────────────────┐
  │  RunnableSequence = conveyor belt   │ → one machine after another
  │  RunnableParallel = two belts side  │ → run together, merge output
  │  RunnablePassthrough = bypass tube  │ → skip transformation, keep original
  │  RunnableLambda = custom machine    │ → your own Python logic inserted
  │  RunnableBranch = sorting gate      │ → route to different belt by rule
  └─────────────────────────────────────┘
        │
        ▼
  Final Product (your output)
```

## Code 1 — RunnableParallel + RunnableSequence (Two Social Posts)

### 🧑‍🏫 What does this code do?

Given a topic, generate a **Tweet** AND a **LinkedIn post** at the same time using two parallel sub-chains.

**Real-world analogy:** You're a social media manager. You give your two copywriters (GPT × 2) the same brief. They work at the same time — one writes a tweet, one writes a LinkedIn post. You collect both when done.

---

### 🔍 Line-by-Line Explanation

```python
from langchain_core.runnables import RunnableSequence, RunnableParallel
```
> Both primitives now live in `langchain_core.runnables` in v1.0+.  
> In v0.3 these were accessed through `langchain.schema.runnable` — that path is **deprecated**.

```python
prompt1 = PromptTemplate(
    template='Generate a tweet about {topic}',
    input_variables=['topic']
)
prompt2 = PromptTemplate(
    template='Generate a Linkedin post about {topic}',
    input_variables=['topic']
)
```
> Two separate prompt templates — same `{topic}` placeholder, different instructions.  
> Both will receive the same input dict `{'topic': 'AI'}`.

```python
model = ChatOpenAI()
parser = StrOutputParser()
```
> One shared model instance. `StrOutputParser` converts `AIMessage → str`.  
> Both sub-chains reuse the same `model` and `parser` objects — perfectly fine.

```python
parallel_chain = RunnableParallel({
    'tweet':    RunnableSequence(prompt1, model, parser),
    'linkedin': RunnableSequence(prompt2, model, parser)
})
```
> **This is the core.** Breaking it down:
>
> `RunnableSequence(prompt1, model, parser)` — explicit sequential chain:
> - Step 1: `prompt1` formats the tweet prompt
> - Step 2: `model` generates the tweet text  
> - Step 3: `parser` extracts plain string
>
> `RunnableParallel({...})` — runs both `RunnableSequence` objects simultaneously, returns:
> ```python
> {'tweet': '🚀 AI is changing...',  'linkedin': 'Excited to share insights on AI...'}
> ```

```python
result = parallel_chain.invoke({'topic': 'AI'})
```
> Sends `{'topic': 'AI'}` to BOTH sub-chains at the same time.  
> Returns a dict with two keys: `'tweet'` and `'linkedin'`.

```python
print(result['tweet'])
print('---')
print(result['linkedin'])
```
> Access each output by its key name from the `RunnableParallel` dict.

---

### 🧠 Mental Model

```
INPUT: {'topic': 'AI'}
        │
        ├─────────────────────┬──────────────────────┐
        │                     │                      │
        ▼                     ▼                      │
 RunnableSequence       RunnableSequence             │
   (tweet branch)        (linkedin branch)           │
        │                     │                      │
  prompt1 → model         prompt2 → model            │
  → parser                → parser                   │
        │                     │                      │
        ▼                     ▼                      │
  "🚀 AI is..."    "Excited to share..."             │
        │                     │                      │
        └─────────────────────┘                      │
                    │                                 │
                    ▼
     {'tweet': '...', 'linkedin': '...'}
```

### ⚠️ v0.3 vs v1.0+ — Code 1

| What | v0.3 | v1.0+ |
|---|---|---|
| Import path | `from langchain.schema.runnable` | ✅ `from langchain_core.runnables` |

```python
    'tweet':    RunnableSequence(prompt1, model, parser),
    'linkedin': RunnableSequence(prompt2, model, parser)
})

result = parallel_chain.invoke({'topic': 'AI'})

print(result['tweet'])
print('-' * 57)
print(result['linkedin'])
```

## Code 2 — RunnableSequence (Joke → Explanation)

### 🧑‍🏫 What does this code do?

Write a joke about a topic, then explain the joke — two model calls in a strict sequence.

**Real-world analogy:** A comedian writes a joke (Step 1), then a teacher explains why it's funny (Step 2). The teacher can't explain the joke before it exists — the order is mandatory.

---

### 🔍 Line-by-Line Explanation

```python
prompt1 = PromptTemplate(
    template='Write a joke about {topic}',
    input_variables=['topic']
)
```
> First prompt — asks GPT to write a joke. `{topic}` is the only input variable.

```python
prompt2 = PromptTemplate(
    template='Explain the following joke - {text}',
    input_variables=['text']
)
```
> Second prompt — asks GPT to explain. Uses `{text}` as the placeholder.  
> **Important:** The variable name is `text`, not `topic`. This matters for what flows into it from step 1.

```python
chain = RunnableSequence(prompt1, model, parser, prompt2, model, parser)
```
> **All 6 steps in one `RunnableSequence` constructor.** Steps run left to right:
>
> | Step | Runnable | Input | Output |
> |------|----------|-------|--------|
> | 1 | `prompt1` | `{'topic':'AI'}` | Formatted prompt string |
> | 2 | `model` | Formatted string | `AIMessage` with joke |
> | 3 | `parser` | `AIMessage` | Plain `str` joke text |
> | 4 | `prompt2` | `str` → fills `{text}` | Formatted explanation prompt |
> | 5 | `model` | Explanation prompt | `AIMessage` with explanation |
> | 6 | `parser` | `AIMessage` | Final plain `str` |

> ⚠️ **Key insight:** At Step 4, `prompt2` receives a plain `str` from `parser`.  
> How does it know to put it in `{text}`? Because when a `PromptTemplate` receives a plain `str` (not a dict), it assumes it's the value for the first — and only — `input_variable`.

```python
print(chain.invoke({'topic': 'AI'}))
```
> Single `.invoke()` call triggers all 6 steps automatically.

---

### 🧠 Mental Model

```
INPUT: {'topic': 'AI'}
        │
        ▼  Step 1
   prompt1 → "Write a joke about AI"
        │
        ▼  Step 2
   model   → AIMessage("Why did the AI break up with...")
        │
        ▼  Step 3
   parser  → "Why did the AI break up with..."  (plain str)
        │
        ▼  Step 4
   prompt2 → "Explain the following joke - Why did the AI..."
        │
        ▼  Step 5
   model   → AIMessage("This joke plays on the idea that...")
        │
        ▼  Step 6
   parser  → "This joke plays on the idea that..."
        │
        ▼
OUTPUT: Full explanation of the joke
```

---

### ⚠️ v0.3 vs v1.0+ — Code 2

| What | v0.3 | v1.0+ |
|---|---|---|
| Multi-step pipeline | `SequentialChain` with named outputs | ✅ `RunnableSequence(a,b,c,d,e,f)` or `a\|b\|c\|d\|e\|f` |
| `SimpleSequentialChain` | ✅ Used for single I/O chains | ❌ Deprecated → use `RunnableSequence` |
| `SequentialChain` | ✅ Used for named I/O | ❌ Deprecated → use `RunnableSequence` with dicts |
| Preferred modern style | `LLMChain` stacks | ✅ LCEL pipes `\|` |

---

### ✅ Full Corrected Code

```python

from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableSequence
from dotenv import load_dotenv

load_dotenv()

    template='Write a joke about {topic}',
    input_variables=['topic']
)

    template='Explain the following joke - {text}',
    input_variables=['text']
)

model = ChatOpenAI()
parser = StrOutputParser()

chain = RunnableSequence(prompt1, model, parser, prompt2, model, parser)

# Equivalent LCEL style (both work identically):
# chain = prompt1 | model | parser | prompt2 | model | parser

print(chain.invoke({'topic': 'AI'}))
```

## Code 3 — RunnablePassthrough (Keep Original + Add New)

### 🧑‍🏫 What does this code do?

Generate a joke, then run two things in parallel:
1. **Keep the joke unchanged** (`RunnablePassthrough`)
2. **Explain the joke** (`RunnableSequence`)

Return both together: `{'joke': '...', 'explanation': '...'}`.

**Real-world analogy:** A newspaper editor receives an article draft.  
- One copy goes to the archive unchanged  
- Another copy goes to the editor for annotation  
- Both copies come back — the original draft AND the annotated version

---

### 🔍 Line-by-Line Explanation

```python
from langchain_core.runnables import RunnableSequence, RunnableParallel, RunnablePassthrough
```
> `RunnablePassthrough` is the new import here. It passes whatever it receives straight through — no transformation.

```python
joke_gen_chain = RunnableSequence(prompt1, model, parser)
```
> First mini-chain. Takes `{'topic': 'cricket'}`, returns a plain string joke.  
> This string becomes the input to the next step.

```python
parallel_chain = RunnableParallel({
    'joke':        RunnablePassthrough(),
    'explanation': RunnableSequence(prompt2, model, parser)
})
```
> **This is the key pattern.** Both branches receive the same input — the plain joke string from `joke_gen_chain`.
>
> `'joke': RunnablePassthrough()` — does nothing. The joke string flows through untouched. Output = same joke string.
>
> `'explanation': RunnableSequence(prompt2, model, parser)` — takes the joke string, passes it into `prompt2` as `{text}`, gets the explanation.
>
> Final output of `parallel_chain`:
> ```python
> {
>   'joke': 'Why did the cricket ball go to school?...',
>   'explanation': 'This joke works because...'
> }
> ```

```python
final_chain = RunnableSequence(joke_gen_chain, parallel_chain)
```
> Chains the two stages:
> - Stage 1: `joke_gen_chain` → generates the joke string
> - Stage 2: `parallel_chain` → forks into passthrough + explanation, merges into dict

```python
print(final_chain.invoke({'topic': 'cricket'}))
```
> Returns the full dict with both `'joke'` and `'explanation'` keys.

---

### 🧠 Mental Model — Why RunnablePassthrough Matters

```
PROBLEM without RunnablePassthrough:
  joke_gen_chain → "Why did the cricket..."
                          │
                          ▼
                    only explanation chain runs
                          │
                          ▼
                   {'explanation': '...'}
                         ❌ joke text is LOST!

SOLUTION with RunnablePassthrough:
  joke_gen_chain → "Why did the cricket..."
                          │
            ┌─────────────┴──────────────┐
            │                            │
            ▼                            ▼
   RunnablePassthrough()          prompt2 | model | parser
   "Why did the cricket..."       "This joke works because..."
            │                            │
            └─────────────┬──────────────┘
                          │
                          ▼
            {'joke': 'Why did...', 'explanation': 'This joke...'}
            ✅ BOTH values preserved!
```

---

### 💡 When to Use RunnablePassthrough

Use it any time you want to **fork** a chain and preserve the value at the fork point:

```
Before fork:   single value (joke text)
After fork:    {'original': original_value, 'transformed': new_value}
```

This is extremely useful for:
- Keeping the original question alongside the answer
- Keeping raw text alongside its summary
- Keeping input data alongside extracted metadata

---

### ⚠️ v0.3 vs v1.0+ — Code 3

| What | v0.3 | v1.0+ |
|---|---|---|
| `RunnablePassthrough` | ❌ Not available | ✅ `from langchain_core.runnables` |
| Preserving original input | Required manual workarounds | ✅ `RunnablePassthrough()` handles it |
| `RunnablePassthrough.assign()` | ❌ Not available | ✅ Add extra keys without losing input dict |

**Bonus — `RunnablePassthrough.assign()` in v1.0+:**
```python
# Instead of building a parallel chain manually,
# you can "assign" new keys to the passthrough:
chain = joke_gen_chain | RunnablePassthrough.assign(
    explanation=RunnableSequence(prompt2, model, parser)
)
# Output: {'joke': '...', 'explanation': '...'}  ← cleaner syntax!
```

## Code 4 — RunnableLambda (Custom Python Function in Chain)

### 🧑‍🏫 What does this code do?

Generate a joke, then in parallel:
1. **Keep the joke** (passthrough)
2. **Count its word count** using a plain Python function

**Real-world analogy:** A proofreader gets a document. One desk keeps the document intact. Another desk runs it through a word-counter tool. Both results come back together.

---

### 🔍 Line-by-Line Explanation

```python
def word_count(text):
    return len(text.split())
```
> A plain Python function. Takes a string, splits it by whitespace, returns the count.  
> This is NOT a LangChain object — yet. We need to wrap it.

```python
joke_gen_chain = RunnableSequence(prompt, model, parser)
```
> Generates the joke as a plain string.

```python
parallel_chain = RunnableParallel({
    'joke':       RunnablePassthrough(),
    'word_count': RunnableLambda(word_count)
})
```
> **`RunnableLambda(word_count)`** wraps the Python function so it behaves like any other LangChain runnable — it can be placed in a sequence, parallel branch, or chain.
>
> Both branches receive the same joke string:
> - `'joke'` → unchanged (passthrough)
> - `'word_count'` → passes through `word_count()` → returns an integer
>
> Output: `{'joke': 'Why did the...', 'word_count': 14}`

```python
final_chain = RunnableSequence(joke_gen_chain, parallel_chain)
result = final_chain.invoke({'topic': 'AI'})
```
> `result` is now: `{'joke': '...', 'word_count': 14}`

```python
final_result = """{} \n word count - {}""".format(result['joke'], result['word_count'])
print(final_result)
```
> Simple Python string formatting to print both values together.  
> Note: `\n` here is a literal backslash-n inside `"""..."""` — it won't render as a newline.  
> ✅ Fix: use `f"{result['joke']}\nword count - {result['word_count']}"`.

---

### 🧠 Mental Model

```
INPUT: {'topic': 'AI'}
        │
        ▼
   joke_gen_chain  →  "Why did the AI break up..."
        │
        ├──────────────────────┬─────────────────────┐
        │                      │                     │
        ▼                      ▼                     │
  RunnablePassthrough     RunnableLambda             │
  "Why did the AI..."     word_count("Why did...")   │
                          = 8                        │
        │                      │                     │
        └──────────────────────┘                     │
                    │
                    ▼
     {'joke': 'Why did the AI...', 'word_count': 8}
```

---

### 💡 RunnableLambda Use Cases

| Use Case | Lambda Function |
|---|---|
| Word count | `lambda text: len(text.split())` |
| Text cleaning | `lambda text: text.strip().lower()` |
| JSON extraction | `lambda text: json.loads(text)` |
| Logging/debugging | `lambda x: (print(x), x)[1]` |
| Format conversion | `lambda d: d['content']` |
| Any Python logic | Any `def` or `lambda` |

---

### ⚠️ v0.3 vs v1.0+ — Code 4

| What | v0.3 | v1.0+ |
|---|---|---|
| Custom functions in chains | `TransformChain` (complex setup) | ✅ `RunnableLambda(fn)` — one line |
| `TransformChain` | ✅ Available but verbose | ❌ Deprecated → `RunnableLambda` |
| Async support | Manual | ✅ `RunnableLambda` auto-handles `async def` |
| Lambda vs named function | Both work | Both work equally |

```python

    return len(text.split())

    input_variables=['topic']

    'word_count': RunnableLambda(word_count)
})

result = final_chain.invoke({'topic': 'AI'})

# ✅ Fixed string formatting
final_result = f"{result['joke']}\nword count - {result['word_count']}"
print(final_result)
```

## Code 5 — RunnableBranch (Smart Summariser)

### 🧑‍🏫 What does this code do?

Generate a detailed report, then check: **is it longer than 300 words?**  
- If YES → summarise it  
- If NO → return it as-is

**Real-world analogy:** An editor receives a report. If it's too long for the brief, they summarise it. If it's already short enough, they leave it alone. The rule is automatic — no manual checking needed.

---

### 🔍 Line-by-Line Explanation

```python
from langchain_core.runnables import RunnableSequence, RunnableParallel, RunnablePassthrough, RunnableBranch, RunnableLambda
```
> All five primitives imported together. Shows the full Runnable family.

```python
prompt1 = PromptTemplate(
    template='Write a detailed report on {topic}',
    input_variables=['topic']
)
prompt2 = PromptTemplate(
    template='Summarize the following text \n {text}',
    input_variables=['text']
)
```
> `prompt1` asks for a long detailed report.  
> `prompt2` is only used if the report is too long — it asks for a summary.

```python
report_gen_chain = prompt1 | model | parser
```
> LCEL-style chain (equivalent to `RunnableSequence`). Generates the full report as a plain string.

```python
branch_chain = RunnableBranch(
    (lambda x: len(x.split()) > 300, prompt2 | model | parser),
    RunnablePassthrough()
)
```
> **The smart gate.** Breaking it down:
>
> `RunnableBranch` checks conditions in order from top to bottom.  
> Each condition is a tuple: `(condition_function, chain_to_run)`.
>
> - Condition: `lambda x: len(x.split()) > 300`  
>   `x` here is the report string. If it has more than 300 words → `True`  
>   → run `prompt2 | model | parser` → returns a summary
>
> - Default (no tuple): `RunnablePassthrough()`  
>   If the report is ≤ 300 words → return it unchanged
>
> **Notice:** `RunnablePassthrough()` is used as the default fallback here — not `RunnableLambda`. This is correct because we want the string returned exactly as-is when it's short enough.

```python
final_chain = RunnableSequence(report_gen_chain, branch_chain)
```
> Stage 1: Generate the report  
> Stage 2: Check length → summarise OR pass through

```python
print(final_chain.invoke({'topic': 'Russia vs Ukraine'}))
```
> A topic like "Russia vs Ukraine" will almost certainly produce a report > 300 words, so the summary branch will fire.

---

### 🧠 Mental Model

```
INPUT: {'topic': 'Russia vs Ukraine'}
        │
        ▼
   report_gen_chain
   (prompt1 | model | parser)
        │
        ▼
   "Russia's invasion of Ukraine began in February 2022...
    [600 words of detailed text]"
        │
        ▼
   RunnableBranch checks:
   ┌──────────────────────────────────────────────────┐
   │  len(x.split()) > 300?                           │
   │  len = 600 → YES ✅                              │
   │  → run (prompt2 | model | parser)                │
   └──────────────────────────────────────────────────┘
        │
        ▼
   "Summarize the following text: Russia's invasion..."
        │
        ▼
   model → 5-line summary
        │
        ▼
   parser → final summary string
        │
        ▼
OUTPUT: "1. Russia invaded Ukraine in Feb 2022..."


─────── IF report was SHORT (< 300 words) ──────────
        │
        ▼
   RunnableBranch checks:
   ┌──────────────────────────────────────────────────┐
   │  len(x.split()) > 300?                           │
   │  len = 150 → NO ❌                               │
   │  → RunnablePassthrough()                         │
   └──────────────────────────────────────────────────┘
        │
        ▼
OUTPUT: [original report unchanged]
```

---

### 💡 RunnableBranch vs if/else in Python

```python
# ❌ Python if/else OUTSIDE the chain:
result = report_gen_chain.invoke({'topic': 'x'})
if len(result.split()) > 300:
    final = (prompt2 | model | parser).invoke(result)
else:
    final = result

# ✅ RunnableBranch INSIDE the chain:
branch = RunnableBranch(
    (lambda x: len(x.split()) > 300, prompt2 | model | parser),
    RunnablePassthrough()
)
final_chain = report_gen_chain | branch
final = final_chain.invoke({'topic': 'x'})
```

The `RunnableBranch` approach is better because:
- The whole pipeline is one object you can `stream()`, `batch()`, or `ainvoke()`
- The graph visualization shows the branch
- The logic is composable and reusable

## LCEL Pipe `|` vs Explicit Runnables Cheat Sheet

These two styles are **100% equivalent** — use whichever is more readable:

```python
# ── Style A: Explicit RunnableSequence (verbose, clear) ──────────
from langchain_core.runnables import RunnableSequence

chain = RunnableSequence(prompt, model, parser)

# ── Style B: LCEL pipe (concise, preferred) ───────────────────────
chain = prompt | model | parser


# ── Parallel: both styles ─────────────────────────────────────────
from langchain_core.runnables import RunnableParallel

# Explicit
parallel = RunnableParallel({'a': RunnableSequence(p1, m, parser),
                             'b': RunnableSequence(p2, m, parser)})

# LCEL dict shorthand
parallel = RunnableParallel({'a': p1 | m | parser,
                             'b': p2 | m | parser})


# ── Passthrough: use when you need to fork and keep original ──────
from langchain_core.runnables import RunnablePassthrough

parallel = RunnableParallel({
    'original':    RunnablePassthrough(),
    'transformed': some_chain
})

# Or cleaner with .assign()
chain = some_chain | RunnablePassthrough.assign(new_key=another_chain)


# ── Lambda: any Python function ───────────────────────────────────
from langchain_core.runnables import RunnableLambda

chain = some_chain | RunnableLambda(lambda x: x.upper())
chain = some_chain | RunnableLambda(word_count)  # named function


# ── Branch: if/elif/else routing ─────────────────────────────────
from langchain_core.runnables import RunnableBranch

branch = RunnableBranch(
    (condition_fn_1, chain_1),   # if
    (condition_fn_2, chain_2),   # elif
    default_chain                # else (RunnablePassthrough or any chain)
)


# ── All chains support the same invocation API ───────────────────
chain.invoke({'key': 'value'})            # sync, one input
await chain.ainvoke({'key': 'value'})     # async
chain.stream({'key': 'value'})            # streaming
chain.batch([{'key': 'v1'}, {'key': 'v2'}])  # multiple inputs
chain.get_graph().draw_ascii()            # visualise (pip install grandalf)
```
